## 🔴 문제 1 (가장 심각): `renderArticles()`가 매번 전체 DB를 3번 읽는다

기사 목록을 화면에 그릴 때마다 아래 3개의 Firebase 쿼리가 **순서대로 순차 실행**됩니다.
순차이므로 A가 끝나야 B가 시작됩니다.

```
① pinnedArticles 로드  →  ② comments 전체 로드  →  ③ users 전체 로드
```

특히 **② comments 전체** 는 댓글이 많을수록 매우 느립니다.
댓글 개수만 표시하기 위해 **전체 댓글 내용**을 다 내려받고 있기 때문입니다.

### ✅ 해결법: 댓글 수를 기사 데이터에 포함시키기

**기사를 저장할 때** 댓글 수 필드를 추가하고, 댓글 작성 시 증감:

```javascript
// 댓글 저장 함수 안에 추가
async function submitComment(articleId, text) {
    // ... 기존 댓글 저장 코드 ...

    // ✅ 댓글 수 카운터 직접 업데이트 (전체 댓글 로딩 불필요)
    await db.ref(`articles/${articleId}/commentCount`)
        .transaction(count => (count || 0) + 1);
}

// renderArticles()에서 comments 전체 로드 제거
// ❌ 기존: const commentsSnapshot = await db.ref("comments").once("value");
// ✅ 대체: 기사 객체의 a.commentCount 를 그대로 사용
```

> Firebase의 `transaction()`은 서버에서 원자적으로 처리되므로 동시 접속에도 안전합니다.

## 🔴 문제 2: 프로필 사진을 위해 `users` 전체를 매번 내려받는다

`renderArticles()`에서 프로필 사진 캐시가 비어있을 때:

```javascript
// script.js 3074번째 줄 — 렌더링 할 때마다 실행될 수 있음
const usersSnapshot = await db.ref("users").once("value"); // ⚠️ 전체 유저 다운로드
```

유저가 100명이면 100명 데이터를 전부 받아서 그 중 화면에 표시된 5~10명의 사진만 사용합니다.

### ✅ 해결법: 필요한 유저만 개별 요청

```javascript
// 필요한 이메일 목록이 있을 때, 유저 전체가 아닌 uid별로만 요청
const emailToUidCache = window._emailToUidCache || (window._emailToUidCache = new Map());

async function fetchProfilePhotos(emails) {
    const uncached = emails.filter(e => !window.profilePhotoCache.has(e));
    if (uncached.length === 0) return;

    // ✅ 필요한 유저만 병렬로 가져오기
    // (users/uid 구조라면 uid를 미리 알아야 하므로, 기사에 uid도 저장하는 게 best)
    // 임시 방편: 전체 대신 limitToFirst 등 조건 쿼리 활용
    const snap = await db.ref("users").orderByChild("email")
        .startAt(uncached[0]).limitToFirst(uncached.length * 2).once("value");
    
    const data = snap.val() || {};
    Object.values(data).forEach(u => {
        if (u?.email) window.profilePhotoCache.set(u.email, u.profilePhoto || null);
    });
    
    // 여전히 없는 건 null 처리
    uncached.forEach(e => {
        if (!window.profilePhotoCache.has(e))
            window.profilePhotoCache.set(e, null);
    });
}
```

> **가장 좋은 방법**: 기사를 저장할 때 `authorUid`를 함께 저장하고,
> `db.ref("users/" + uid + "/profilePhoto").once("value")` 로 해당 유저만 요청합니다.

## 🔴 문제 3: 채팅 목록 로딩 시 `users` 전체를 또 내려받는다

`loadChatRoomList()` 함수 (chat.js 325번째 줄):

```javascript
const [chatSnaps, usersSnap, savedOrder] = await Promise.all([
    Promise.all(myRoomIds.map(id => getChatDb().ref(`chats/${id}`).once('value'))),
    db.ref('users').once('value'),  // ⚠️ 모든 유저 전체 로드
    loadChatRoomOrder(mainUid)
]);
```

채팅 상대방의 닉네임/사진을 보여주기 위해 전체 유저를 내려받고 있습니다.

### ✅ 해결법: 채팅방에 상대방 정보를 미리 저장

채팅방을 만들 때 상대방 닉네임과 사진을 채팅방 데이터에 포함시키면
유저 전체를 로드할 필요가 없어집니다.

```javascript
// 채팅방 생성 시 상대방 정보 포함
const roomData = {
    participants: { [myUid]: true, [otherUid]: true },
    // ✅ 표시용 정보를 미리 저장
    participantInfo: {
        [myUid]: { nickname: myNickname, photo: myPhoto },
        [otherUid]: { nickname: otherNickname, photo: otherPhoto }
    },
    lastMessageAt: Date.now(),
};
```

그러면 `loadChatRoomList()`에서 `db.ref('users')` 호출을 완전히 제거할 수 있습니다.

## 🟡 문제 4: `pinnedArticles`를 매번 렌더링마다 다시 로드한다

`renderArticles()`가 호출될 때마다:

```javascript
const pinsSnapshot = await db.ref("pinnedArticles").once("value"); // 매번 요청
```

고정 기사는 관리자가 바꾸기 전까지 바뀌지 않는 데이터입니다.

### ✅ 해결법: 캐시 적용 또는 실시간 리스너로 전환

```javascript
// 방법 A: 메모리 캐시 (간단)
let _pinnedCache = null;
let _pinnedCacheTime = 0;

async function getPinnedArticles() {
    const CACHE_TTL = 60_000; // 60초 캐시
    if (_pinnedCache && Date.now() - _pinnedCacheTime < CACHE_TTL) {
        return _pinnedCache;
    }
    const snap = await db.ref("pinnedArticles").once("value");
    _pinnedCache = snap.val() || {};
    _pinnedCacheTime = Date.now();
    return _pinnedCache;
}

// renderArticles() 안에서 교체
// ❌ const pinsSnapshot = await db.ref("pinnedArticles").once("value");
// ✅ const pinnedData = await getPinnedArticles();
```

## 🟡 문제 5: JS 파일 9개가 `defer` 없이 동기 로딩된다

`index.html` 맨 아래:

```html
<!-- ❌ 현재: 순서대로 동기 로딩 (앞 파일이 파싱될 때까지 다음 파일 시작 안 됨) -->
<script src="./js/script.js"></script>
<script src="./js/script2.js"></script>
<script src="./js/ganadi-helper.js"></script>
<script src="./js/category-tutorial.js"></script>
<script src="./js/chat.js"></script>

```

모든 스크립트가 HTML 파싱을 블록하면서 순차 실행됩니다. `script.js`가 10,000줄이므로 파싱에 시간이 걸립니다.

### ✅ 해결법: Quill만 제외하고 defer 적용

```html
<!-- ✅ 개선: 병렬 다운로드 + HTML 파싱 후 실행 -->
<!-- Quill은 script.js가 참조하므로 먼저 로드해야 함 -->
<script src="https://cdn.quilljs.com/1.3.7/quill.js" defer></script>

<script src="./js/script.js" defer></script>
<script src="./js/script2.js" defer></script>
<script src="./js/ganadi-helper.js" defer></script>
<script src="./js/category-tutorial.js" defer></script>
<script src="./js/chat.js" defer></script>
<script src="./js/vote.js" defer></script>
<script src="./js/Console-toggle.js" defer></script>
<script src="./js/Chat-upgrade.js" defer></script>
<script src="./js/Device-detect.js" defer></script>
```

> `defer`는 실행 순서를 보장하면서 병렬 다운로드를 허용합니다.

## 🟡 문제 6: `<head>`에 폰트/아이콘 preconnect가 없다

```html
<!-- ❌ 현재: 연결 설정 지연 -->
<link href="https://fonts.googleapis.com/..." rel="stylesheet">
<link rel="stylesheet" href="https://cdnjs.cloudflare.com/...font-awesome...">
```

### ✅ 해결법: `<head>` 최상단에 preconnect 추가

```html
<head>
    <meta charset="UTF-8">
    <!-- ✅ 이 두 줄을 head 가장 위에 추가 -->
    <link rel="preconnect" href="https://fonts.googleapis.com">
    <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
    <link rel="preconnect" href="https://cdnjs.cloudflare.com">
    
    <!-- 나머지 기존 코드 ... -->
</head>
```

## 📊 예상 개선 효과 요약

| 문제 | 현재 소요 | 개선 후 | 개선율 |
|------|---------|--------|--------|
| 문제 1: comments 전체 로드 | 댓글 수×크기 비례 | 0 (저장된 카운트 사용) | ~100% 제거 |
| 문제 2: users 전체 로드 (기사) | 전체 유저 크기 | 필요한 유저만 | ~80% 감소 |
| 문제 3: users 전체 로드 (채팅) | 전체 유저 크기 | 0 (채팅방에 포함) | ~100% 제거 |
| 문제 4: pinnedArticles 반복 | 매 렌더마다 요청 | 60초 캐시 | ~95% 감소 |
| 문제 5: JS 동기 로딩 | 순차 파싱 블록 | 병렬 다운로드 | 체감 빠름 |
| 문제 6: 폰트 연결 지연 | DNS lookup 포함 | preconnect 선점 | 50~200ms 감소 |

## 🛠️ 적용 우선순위 권장

```
1순위 (효과 최대): 문제 1 — commentCount를 기사에 포함 (코드 수정 필요)
2순위 (즉시 적용 가능): 문제 5, 6 — defer + preconnect (index.html 수정만)
3순위: 문제 4 — pinnedArticles 캐시 (함수 하나 수정)
4순위: 문제 2, 3 — users 쿼리 최적화 (구조 변경 필요)
```

1순위와 2순위만 적용해도 **첫 로딩 및 기사 목록 로딩 속도가 눈에 띄게 빨라질 것**입니다.
